# 系统规格 09：UI、UX 与可访问性

## 1. 设计目标

复杂不等于隐晦。玩家应能回答：发生了什么、为什么、我能做什么、代价是什么、何时生效。

## 2. 交互原则

- 玩家处理例外、任命和战略，不逐地区重复点击。
- 同时支持“先选对象再选命令”和“先选命令再筛可用对象”。
- 批量操作是领域层命令，提交前有预览与部分失败说明。
- 所有预测显示区间、置信度和前三大因素。
- 关键数值提供反事实提示：改变什么会改善结果。
- 历史术语先给白话功能，再给术语与背景。

## 3. 主界面

- 中央：Pixi 战略地图。
- 顶栏：日期、季节、速度、核心资源、紧急状态。
- 左/右可折叠面板：选中对象、命令、计划、通知。
- 底部/抽屉：时间线、战报、待决提案。
- 全局搜索/命令面板：人物、地区、势力、行动。

不得把所有功能塞入多层全屏菜单，使玩家失去地图上下文。

## 4. 信息层级

地图缩放 LOD：

- 远景：政体、宗主网络、主要战役、贸易/补给。
- 中景：城、主要地区、军队、道路、风险。
- 近景：地方设施、驻军、人物与细节标签。

地图模式：控制、宗主、补给、贸易、情报、人口、行政负荷、合法性。模式之间共享筛选和图例。

## 5. 列表与批量操作

人物/地区表需要：

- 多列排序；
- 可组合筛选；
- 保存视图；
- 多选；
- 批量命令预览；
- 虚拟化；
- 解释不可执行原因；
- 快捷跳转地图。

更换领主时，既有辖区方针默认保留；界面明确显示新任执行效果差异。

## 6. 战役规划器

必须同时显示：

```text
目标与战后意图
集结日与各军预计到达
安全补给天数
季风窗口
敌军估计与可信度
盟友/臣属履约状态
关键瓶颈
取消条件
```

提供“同步到达”自动建议和可解释冲突，不要求玩家手工试错。

## 7. 通知

分为：

- `DECISION_REQUIRED`：必须玩家处理；
- `RISK_CHANGED`：计划假设已显著变化；
- `COMPLETED`：重要结果；
- `FYI`：可汇总；
- `DEBUG`：开发构建。

同类低重要事件合并，不进行通知轰炸。每条通知可跳转原因对象/计划。

## 8. 响应式与显示

必须测试：1280×720、1280×800、1920×1080、2560×1440、3440×1440、4K 150%/200%。

- DOM UI 不经低分辨率 canvas 缩放。
- UI 缩放范围目标 80%–200%。
- 字号有最小下限。
- 侧栏可折叠，表格列可隐藏。
- 16:10 与超宽不裁切。
- Retina/高 DPI 地图和文字分别处理。

## 9. 输入与可访问性

首发优先鼠标键盘：多选、右键、快捷键、悬停。手柄为后续独立密度/导航方案，不强迫 PC UI 巨型化。

要求：

- 全部核心流程键盘可达；
- 焦点清晰，弹窗正确捕获/恢复；
- 不仅以颜色编码；
- 色觉模式/高对比；
- 动画减弱；
- 可暂停所有时间压力；
- 文字缩放不破坏布局；
- 屏幕阅读器至少覆盖菜单、表格和关键决策，不要求读 Pixi 地图每个装饰。

## 10. AI 开发可测试性

每个复杂 UI 提供 Storybook fixtures 和开发路由，例如：

```text
/dev/ui/campaign-planner?fixture=late-allies
/dev/ui/castle-panel?fixture=no-governor
/dev/map?fixture=500-armies
```

开发构建暴露只读/受控 `window.__GAME_DEBUG__`，供 Playwright 加载 fixture、推进日期、提交命令和读取状态哈希。
