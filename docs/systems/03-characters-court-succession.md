# 系统规格 03：人物、宫廷与继承

## 1. 设计目标

人物是拥有能力、关系、家族利益和地方基础的代理人，不是可任意移动的属性卡。玩家的主要政治体验来自“把权力交给谁”。

## 2. 人物数据

公开能力控制在四项：

- `command`：组织、军心、协同；
- `valor`：局部战斗与威慑；
- `administration`：财政、开发、组织；
- `intrigue`：情报、外交、调略、风险判断。

其他维度：

```text
traits
values
ambition
loyaltyDrivers
kinship
patronClientRelations
personalRivalries
religiousCourtAffiliations
landedPower
merit
health/age
historicityTag
```

能力并非所有结果的线性乘数；职位适配、关系、信息和资源同样重要。

## 3. 关系图

关系使用有方向的边：

```text
trust
affection
fear
debt
rivalry
patronage
kinshipType
```

只保留有游戏意义的关系，避免完整 O(n²) 矩阵。边随事件更新并可衰减。

## 4. 职位与权力

职位包含：

- 权限；
- 管辖对象；
- 可支配收入/兵力；
- 任期或世袭预期；
- 身份/能力限制；
- 功勋和关系后果。

方针属于职位/辖区，不随当前任职者离开而无故清空；执行效果由任职者改变。

## 5. 提案与自主行为

人物在发现机会或问题时生成提案：

```text
发现 → 形成方案 → 估计收益/风险 → 受个性与自身利益修正 → 提交/自行执行
```

玩家看到主要理由，而非完整隐藏权重。高自治职位在授权范围内可直接执行；越权行为可能形成政治事件。

## 6. 功勋与赏罚

功勋来源：作战、治理、外交、救援、履约。奖励可以是：

- 领地/收入权；
- 职位；
- 婚姻/亲近；
- 荣誉与仪式地位；
- 财物；
- 对家族的保障。

奖励不等价。给一名功臣更多土地会提升其满足和能力，也会提升独立基础。

## 7. 继承

继承危机是主系统，不是随机事件弹窗。候选人的支持来自：

```text
kinshipClaim
rulerDesignation
courtSupport
militarySupport
provincialSupport
suzerain/foreignSupport
ritualLegitimacy
hostageStatus
```

继位结果可以是和平继承、摄政、共治、分裂、拥立、内战或臣属脱离。不要假设统一的长子继承规则。

## 8. 死亡与连续性

人物死亡必须触发：

- 职位空缺；
- 既有辖区方针保留但执行力变化；
- 私人关系/债务重新结算；
- 家族与客户端继承；
- 军队指挥替换；
- 必要的继承事件。

不得出现死亡人物继续提案、带兵或作为关系端点参与新计算。

## 9. 不变量

- 一人同一时刻只能位于一个实体位置/军队。
- 互斥主职不能并存。
- 家族边无逻辑循环（例如本人为自己的祖先）。
- 权限来自职位/政体，不从 UI 隐式获得。
- 历史人物标签和来源必须可追溯。

## 10. 垂直切片

至少 50 人物，包含：可信能臣、强势地方领主、野心将领、继承竞争者。玩家更换两种城主后，应在开发、备战、提案和政治风险上看到可解释差异。
