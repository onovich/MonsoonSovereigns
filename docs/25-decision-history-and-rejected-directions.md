# 25. 决策历史与被否决方向

> 本文解释项目为什么成为现在的样子。它不是重新开放所有决定的邀请；冻结状态以 `PROJECT_STATUS.md` 和 ADR 为准。

## 1. 从“类《信长之野望》”得到的机制结论

项目最初目标并不是复刻某一代作品的菜单，而是提炼该类型真正成立的结构：

```text
土地与人口
→ 职位、知行与地方权限
→ 代理人治理和备战
→ 战役、补给、会战与围城
→ 功勋、俘虏、威望和领地变化
→ 赏罚、分封与组织重组
→ 新一轮统治问题
```

从对标作品中保留的设计价值：

- 从《创造》一类空间系统继承“道路、距离、人口、直辖压力和军团自治彼此联动”的思想。
- 从《大志》一类社会经济系统继承“动员方式改变生产结构、商业关系不是单一税收按钮”的思想。
- 从《新生》一类组织系统继承“武将是有动机的代理人，大名主要任命、授权、判断和处理例外”的思想。

明确不复刻：

- 仅用固定城池上限模拟行政能力。
- 让所有领地归属、外交与人物忠诚简化为单一数值。
- 让手操会战完全覆盖动员、粮道与战备。
- 让一次会战自动把大片地区永久翻色。
- 后期要求玩家逐城重复下达同类命令。

因此项目的核心问题被确定为：

> 失去一名可信而能干的地方统治者，是否会迫使玩家改变整个战略？

若答案是否定，人物仍只是属性卡，项目核心没有成立。

## 2. 技术路线的演变

### 2.1 第一版候选：Unity + 纯 C# 模拟核心

Unity 最初被视为务实首选，原因是 Windows、macOS、Web、多媒体和 2.5D 渲染能力成熟；同时计划把模拟核心放在不依赖 Unity 的纯 C# 模块中。

### 2.2 推翻 Unity 的关键前提

用户进一步明确：项目将大量交给 Codex/AI Agent 执行，开发效率和可自动验证性比传统编辑器便利更重要。

Unity 的主要问题并非性能不足，而是大量项目状态存在于场景、Prefab、Inspector、序列化引用、导入设置和编辑器 GUI 中。Agent 很难像处理源码、浏览器 DOM、JSON 和测试日志一样可靠地理解、修改并验证这些状态。

因此技术评价函数发生变化：

```text
传统团队：引擎美术工作流、原生能力、招聘生态权重较高
AI 主导团队：文本可见性、可脚本化、端到端验证、单语言和快速反馈权重更高
```

### 2.3 最终决定：Web-first

冻结为：

```text
TypeScript + React + PixiJS + Dedicated Web Worker
+ Node.js headless runner + Electron Windows shell
```

原因：

- 规则、UI、数据、工具和桌面壳主要使用同一种语言。
- Storybook、Playwright、浏览器 Trace、截图和 DevTools 适合 Agent 闭环验证。
- DOM 适合信息密集型表格和布局，PixiJS 适合地图和高频视觉对象。
- 模拟置于 Worker，既避免阻塞 UI，也形成未来替换实现的清晰协议边界。
- Node runner 与浏览器共用核心，可大规模跑局和做确定性验证。

首版不引入 Rust/WASM；只有 profiler 证明特定纯计算内核是瓶颈，才允许通过 ADR 局部替换。

## 3. 平台决定的演变

最初希望 Windows、macOS 都可玩，Web 最好可玩。重新评价开发与发行成本后，冻结为：

1. Windows Electron：正式桌面平台。
2. Chrome/Edge Web：正式 Web 平台。
3. Firefox：二级支持。
4. macOS 浏览器：尽力兼容。
5. macOS 桌面应用：暂缓，不在架构上封死。
6. Safari、移动端、触屏专用 UI：非首发目标。

暂缓 macOS 桌面版是发布、签名、公证、设备与 QA 成本决定，不是业务代码允许 Windows 专属耦合。平台功能必须经过 `PlatformAdapter`。

## 4. 题材候选与选择

### 4.1 被否决：继续做三国或日本战国

问题不是题材本身质量低，而是：

- 市场和玩家已经形成极强的既有比较框架。
- 人物、城池和事件内容量容易吞噬小团队资源。
- 玩家常默认追求直接吞并与统一，反而削弱“臣属、授权、继承和多层权力”的原创价值。

### 4.2 候选：德干高原

优势：收入权、军役、要塞网络、多来源军队与本项目高度契合。

暂缓原因：制度、语言、宗教、地方史和现代叙事风险很高；第一作需要更强的本地顾问与研究预算。

### 4.3 候选：海峡与香料群岛

优势：差异化最大，可围绕港口、商人、航路、海上群体和香料贸易设计。

暂缓原因：它会要求海战、航路和港口经济成为主骨架，不能只在现有“河谷—地方领主—陆上战役”架构上换皮，第一作范围风险较大。

### 4.4 候选：明代西南土司地区

优势：中央册封、地方世袭、贡赋兵役和多层权威高度匹配。

暂缓原因：现代族群议题敏感，史料视角不均，需更强本地研究与文化审查。

### 4.5 候选：季风低奇幻

优势：可以自由组合最佳机制并降低具体历史还原压力。

暂缓原因：必须自行建立数百人物、地名、王朝和文化的认知价值；世界观与美术生产成本可能高于真实历史题材，而且容易落入泛亚洲拼贴。

### 4.6 最终选择：约 1531—1600 年东南亚大陆

选择它不是为了“冷门”，而是因为历史结构天然支持：

- 河谷中心与山地/边缘政体并存。
- 贡赋、臣属、婚姻、人质与地方王族保留。
- 季风、河运、稻作周期和战争窗口。
- 快速扩张的霸权与继承后迅速瓦解。
- 人口、工匠、军役与通路比现代边界面积更重要。

产品一句话：

> 在季风决定战争、人口重于土地的时代，通过任命、贡赋、婚姻、人质与征服，让诸王承认并持续服从你的霸权。

## 5. 题材选择带来的系统变化

这不是日本战国换皮。基础语法从：

```text
攻城 → 占领 → 直属领土 → 继续扩张
```

变为：

```text
获得军事优势
→ 谈判臣服方式与义务
→ 保留或替换地方统治结构
→ 处理人质、驻军、贡赋、出兵和继承
→ 在中央能力下降时重新争取服从
```

因此：

- “地区实际控制”与“宗主关系、义务、宣称、礼仪承认”分层建模。
- 胜利条件包含霸权、合法性、贸易与长期延续，不强制地图全涂色。
- 战象是指挥、士气与威望单位，不是简单重型坦克。
- 强制迁徙、俘虏、奴役和人口掠夺不能作为无代价正收益表现。

## 6. 后续更改规则

出现下列情况才能重开上述决定：

- 垂直切片用数据证明核心体验不可成立。
- 平台性能测试证明正式目标无法满足。
- 一手/高质量研究证明世界模型存在根本错误。
- 商业或法律条件发生足以改变可行性的变化。

更改必须包含：新 ADR、原决定、证据、替代方案、迁移成本、存档/内容影响、独立审查和明确批准。不得以“实现更方便”为由在普通 PR 中偷换产品方向。
