# 对标机制分析：从《信长之野望》提取什么，不复制什么

> 本文记录项目最初形成时的参考结论。原作现代商业源码未公开；机制判断来自官方说明书/开发资料，社区公式只作实测线索，不冒充源码。

## 1. 对标的真正核心

类《信长之野望》的体验不是“经营菜单 + 即时战斗”，而是：

```text
土地与人口
→ 知行、职位与权限
→ 家臣进行治理、动员、调略和备战
→ 统治者批准目标、任命和资源
→ 行军、补给、会战、攻城
→ 领土、威望、俘虏和功勋变化
→ 赏罚、重新分封与组织扩张
→ 新的政治风险
```

本项目把它重写为臣属、义务和季风网络，而非换皮复制日本战国。

## 2. 《创造PK》的可取部分

### 机制骨架

- 城—道路—人口—劳力—兵力形成空间经济；
- 距离和道路影响行军，也限制直接统治；
- 超出直辖能力的地区交给军团；
- 军团根据攻略目标和内政方针自主执行；
- 路线、夹击、军粮和侦察让地图拓扑有意义。

### 本项目的转换

- 用图结构统一道路、河运、贸易、补给、情报和行政通信；
- 用渐进行政负荷替代二元直辖上限；
- 用职位/臣属权限替代纯自动化军团；
- 修路既改善统治和贸易，也可能帮助敌军深入。

### 要避免

- 把操作上限做成人为不可解释限制；
- 后期只剩批量设定相同城方针；
- AI 军团行为不透明。

## 3. 《大志PK》的可取部分

### 机制骨架

- 农业按季节、劳力、水利和人口组织；
- 商圈允许共同进入、投资、分享或垄断；
- 农兵占用农业人口，职业足轻持续消耗现金；
- “志”和方策可形成不同经济—军队转换结构。

### 本项目的转换

- 动员农户影响后续农时和收获；
- 常备随从稳定但造成长期财政与政治成本；
- 市场依赖路线、安全、信用和商人选择，不因占城自动获得；
- 政策改变转换规则，而不是堆百分比。

### 要避免

- 季节结算变成重复表格劳动；
- 资源种类膨胀；
- “理念”只是一条最优数值树。

## 4. 《新生PK》的可取部分

### 机制骨架

- 郡、领主、城主、身份和知行构成多层代理；
- 领主会自动开发和在能力范围内调略；
- 城主能力受属下和关系影响；
- 攻略目标触发备战，不是即时凭空出兵；
- 节点式会战强调据点、夹击、退路、士气与统帅；
- 家臣提案并不永远最优，玩家判断成本、风险和提出者。

### 本项目的转换

- 人物是有自身家族和地方基础的代理人；
- 方针归辖区/职位，执行力归人物，避免换人后设置消失；
- 战役规划明确集结日、补给、季风和臣属履约；
- AI 和提案输出理由码与 Decision Trace；
- 重大胜利只造成周边动摇，不自动翻转领土。

### 要避免

- 自治太强导致玩家无事可做；
- 自治太弱只剩批量自动按钮；
- 玩家不知道军团为何不动；
- 手动会战压倒战略准备。

## 5. 社区常见技术/UX 抱怨转成约束

| 抱怨样本 | 项目约束 |
|---|---|
| 固定 16:9、16:10 裁切、文字缩放糊 | DOM 响应式 UI，独立高 DPI，分辨率矩阵测试 |
| 主机式巨大按钮牺牲 PC 信息密度 | 鼠标键盘优先，手柄后续独立布局 |
| 逐城设置、换领主后方针丢失 | 批量命令是一等领域命令；方针属于职位/辖区 |
| 军团长期不动且无解释 | Decision Trace、阻塞 reason code、计划预测 |
| 部队到达时间需手工逐支调整 | 集结日与同步到达规划器 |
| 每个对象必须反复点开 | 全局检索、表格、命令优先筛选和保存视图 |

## 6. 最终组合

```text
《新生》的分封/代理人
+《大志》的人口—军队与商贸转换
+《创造》的道路、空间行政和军团骨架
+ 本项目独有的季风、臣属义务、继承和人口代价
```

不是把三代所有按钮拼接。每个系统必须回答同一个问题：

> 玩家如何分配权力来组织战争，又如何承担分配权力的后果？
