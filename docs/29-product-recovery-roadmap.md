# 《季风诸王》产品重基线 Roadmap

> 状态：ACCEPTED
>
> 日期：2026-07-10
>
> 产品所有者批准：当前对话中的“可以，接受”明确决定
>
> 适用范围：当前 M7 战略地形恢复之后的产品、技术、内容与发布规划
>
> 权威关系：本文件保留既有 Gate 历史，但从本批准日起成为前向产品成熟度 Roadmap；`docs/11-roadmap.md` 和 `docs/23-milestone-acceptance-matrix.md` 必须在 R0 中同步重基线。

## 1. Roadmap 目的

本 Roadmap 用于纠正当前项目中“工程任务大量关闭，但玩家产品尚未形成真实闭环”的成熟度偏差。

它遵循以下原则：

1. 保留 M0-M6 已验证的确定性模拟、协议、存档、内容管线、AI 和平台工程成果，不重新初始化或重做已证明正确的底层能力。
2. 不再把无头 Runner、静态 fixture、客户端本地步骤或 DOM 属性视为玩家可玩性的替代证据。
3. Web 客户端中的玩家操作必须经过真实 Worker、Command、Query、Read Model 和 Save/Load 边界。
4. 地图必须首先是一张可理解、可操作的战略地形图，然后才是数据结构和渲染技术演示。
5. 内容扩张、美术完整化和正式内容锁必须晚于核心产品闭环。
6. Roadmap 以退出条件管理，不在缺少团队容量基线时伪造日期或工期。

## 2. 当前事实基线

### 2.1 可保留资产

下列成果继续作为有效工程资产：

- 确定性 `sim-core`、固定 GameDay 调度和状态 Hash；
- 玩家与 AI 共用的 Command 模型；
- 显式 Query / Read Model、协议 schema 和拒绝语义；
- M1-M6 规则系统、无头场景、AI、存档和回放能力；
- React DOM、PixiJS、Dedicated Web Worker、Web-first 和 Windows Electron 的冻结技术方向；
- 内容来源、历史 claim、文化审查和本地化管线；
- pnpm monorepo、TypeScript strict、Vitest、Playwright、Storybook 和 CI 基础设施。

### 2.2 不得继续沿用的成熟度假设

下列说法不能继续作为产品阶段结论：

- “Runner 能从开局跑到胜利”不等于玩家客户端可从开局玩到胜利；
- “UI 能生成 Command DTO”不等于 Command 已被权威模拟接受；
- “界面显示提交成功”不等于世界状态已经改变；
- “E2E 点击了预览和确认”不等于真实运行时闭环成立；
- “地图没有矩形格”不等于地图已经具备连续地形和战略可读性；
- “已有平台构建”不等于产品达到了 Alpha、Beta 或 Release Candidate。

### 2.3 当前产品成熟度判断

当前项目应定义为：

```text
确定性模拟原型：成熟
规则与数据能力：广泛但需集成验证
Web 产品运行时：未闭环
核心交互体验：原型阶段
战略地图：恢复阶段
内容生产：不应继续锁定
发行准备：尚未开始
```

既有 M0-M6 Gate 保留为历史工程证据。当前及后续产品成熟度由本文件的 R0-R8 Gate 重新判断。

## 3. 成熟度术语

### 3.1 Playable

只有同时满足以下条件，才能称为“可玩”：

- 玩家能理解当前身份、时间、局势和至少一个可执行目标；
- 玩家输入通过权威 Command 路径改变 WorldState；
- 界面显示模拟返回的接受、拒绝、代价和结果；
- 时间可以暂停、推进，并触发真实日/月/季节结算；
- 存档后重启客户端可以恢复相同权威状态；
- 不依赖测试 fixture、预写结果或仅存在于 React 本地的流程状态。

### 3.2 Alpha

Alpha 表示主要系统在玩家客户端中形成从开局到至少一种胜利或失败的完整闭环。允许内容少、平衡粗、视觉低保真，但不允许主要流程是占位实现。

### 3.3 Beta

Beta 表示主要系统和玩家流程已经锁定，内容达到目标规模，后续只允许缺陷修复、明确调平、性能修复和经过批准的内容修订。

### 3.4 Release Candidate

Release Candidate 表示候选构建可以按真实发布方式安装、更新、卸载、保存、恢复和回滚，并通过安全、许可证、兼容性、人工通关和发布长跑检查。

## 5. R0 - Recovery Baseline

### 目标

建立一份可以被产品、工程和 QA 共同信任的当前状态，停止在不可靠基线上继续叠加功能。

### 交付

- 将当前未完成的 M7 地图分支收敛为可审查的保留、拆分或放弃决定；
- 清点并关闭失效 agent、worktree、branch 和 route 状态，但保留历史记录；
- 将 `PROJECT_STATUS.md` 压缩为当前阶段、当前任务、最近 Gate、风险和下一动作；
- 将历史里程碑记录移入独立 history/ledger，而不是继续追加到当前状态文件；
- 建立生产入口与 fixture/test 入口清单；
- 建立“底层能力已存在但客户端未接入”的能力矩阵；
- 冻结 R1 以前的内容扩张、内容锁、美术生产和非必要本地化填充。

### 退出 Gate

- `origin/main`、任务图、当前状态和实际工作树一致；
- 没有伪 active agent 或未知 worktree；
- 所有生产入口、fixture 和 no-op command 路径已登记；
- 当前分支不存在无法归属的业务改动；
- 产品所有者接受本 Roadmap 或给出明确修订。

### 非目标

- 不在 R0 重写模拟系统；
- 不在 R0 制作新地图、美术或内容；
- 不用整理流程代替产品修复。

## 6. R1 - Authoritative Client Runtime

### 目标

让 Web 客户端成为真实游戏宿主，而不是静态 fixture 展示器。

### 交付

- 正式 Simulation Web Worker 启动、关闭、重连和错误恢复；
- Boot Scenario、Query、Preview Command、Submit Command、Read Model Delta、Save 和 Load 的完整协议往返；
- 主线程不直接调用 `sim-core`；
- 生产入口不加载 M3-M7 演示 fixture；
- M3、M4、M5、M6 的 no-op command handler 被真实 Client Runtime 替代；
- 权威时间控制：暂停、1x、2x、4x 和自动暂停；
- Worker 返回的接受、拒绝和结果驱动 UI 状态；
- Web 存储适配器与 Electron 文件适配器使用同一存档格式；
- Worker 崩溃后可以从最近存档或明确错误状态恢复。

### 强制验收场景

1. 启动场景并取得初始 Read Model；
2. 预览一个合法命令并确认代价；
3. 提交命令，WorldState Hash 改变；
4. 提交非法或过期命令，世界状态不改变且原因可见；
5. 推进至少一个月度结算；
6. 保存、重启页面、载入并恢复相同 Hash；
7. Node Runner 与浏览器对同一存档和命令尾得到相同结果。

### 退出 Gate

- 生产路径中没有 `void command`、预写成功文本或客户端伪终局；
- Playwright 操作真实 Worker，而不是注入结果 fixture；
- 浏览器和 Node 确定性、存档 round-trip、Worker restart 全部通过；
- 至少一次人工操作证明世界状态发生可解释变化；
- 无 P0/P1。

## 7. R2 - First Playable Court Loop

### 目标

用一个小型复合场景证明“观察、决定、推进、反馈”是清晰且有趣的。

### 玩家循环

```text
月初宫廷评议
-> 选择一项主要压力
-> 预览候选方案和代价
-> 下达任命、义务或政策命令
-> 推进时间
-> 观察地方、财政、忠诚和准备度变化
-> 月末总结与下月优先事项
```

### 交付

- 清晰的月度节奏、当前阶段和时间推进控制；
- 首屏只突出身份、时间、主要压力和建议下一步；
- 义务处理包含履行、部分履行、延期、拒绝和减免的真实结果；
- 任命流程包含候选差异、预览、确认、结果和不可重复提交状态；
- 至少一个顾问提案，但提案不保证最优；
- 所有操作显示前三项影响因素和一个可执行改进建议；
- 结果反馈同时出现在地图、任务队列和相关对象 Inspector；
- 新手可以在不阅读开发说明的情况下完成第一个月。

### 退出 Gate

- 至少 3 名非实现者完成首次月度循环；
- 没有人需要开发者指出“按钮在哪里”才能处理首项义务；
- 每个确认动作都有明确的处理中、接受、拒绝或完成状态；
- 重复点击、过期预览、错误主体和资源不足均被稳定处理；
- 人工记录确认核心循环可理解，无 P0/P1 交互阻断。

## 8. R3 - Strategic World Map

### 目标

建立符合大战略游戏阅读习惯的战略地形地图，使玩家能够从地图上理解地貌、通道、政治控制、风险和行动目标。

### 权威模型

必须分离以下概念：

- 世界坐标与投影；
- 连续地形底图和地貌区域；
- 河流、海岸、山脊、湿地等屏障与通道；
- 城市、港口、关隘、渡口、仓库等战略节点；
- 道路、河运、山道和沿海航线构成的路线图；
- 地区治理和政治控制覆盖层；
- 渲染分块、LOD 和空间索引。

路线可达性由权威路线图决定。治理多边形、视觉邻近、质心距离和渲染线条不得创造路线。

### 技术交付

- 明确地图投影和坐标基准；
- 连续地形底图，不以治理多边形伪装地形；
- 路线图与政治覆盖层独立；
- R-tree、quadtree 或等价分块索引用于可见对象和命中查询；
- 节点、路线、标签和覆盖层具有稳定缩放级别；
- Pixi Renderer 响应 ResizeObserver、DPI 和窗口变化；
- 点击、框选、hover 和路线预览使用同一空间查询语义；
- 地图可从当前焦点直接发起义务、任命、动员或战役相关操作。

### 视觉验收

- 不显示规则网格、蜂窝伪装或密铺治理块作为主要地貌；
- 玩家在 10 秒内指出主河谷、主要山系/屏障、关键城市/港口和当前战略目标；
- 路线、地形、政治控制和任务风险有清晰但不相互覆盖的视觉层级；
- 1280x720、1440x900、1920x1080、150% DPI 下地图都占据主要决策空间；
- 截图不得通过测试专用 CSS 改变产品布局后再取证；
- 最终接受包含产品所有者人工检查。

### 非目标

- 不要求正式 GIS 数据或最终历史边界；
- 不要求 3D 地形、手动会战或生产级美术；
- 不以增加多边形数量冒充地理真实性。

## 9. R4 - Integrated Campaign Loop

### 目标

把已经存在的 M2-M4 模拟能力接入真实玩家流程，完成一场从准备到战后治理的短战役。

### 交付

- 选择战争目标和政治目的；
- 预览集结、兵力、粮食、路线、季风窗口和机会成本；
- 动员、准备、出征、行军、补给、援军、自动会战和围城；
- 撤军、投降条件和战后直接统治/分封/臣属安排；
- 农时、财政、劳力、忠诚和信用受到真实反馈；
- AI 使用同一 Command 路径并显示等待、改目标和撤军理由；
- 战役过程中可保存、退出和恢复。

### 退出 Gate

- 玩家可以在客户端完成、取消或失败一场短战役；
- 战役结果真实进入 M3 治理状态，而不是展示独立战报 fixture；
- 雨季、断粮、路线阻塞和援军迟到至少各有一个负向场景；
- 玩家与 AI 的命令权限和原因语义一致；
- 同一战役可以重放并得到相同结果；
- 无 P0/P1。

## 10. R5 - Product Alpha

### 目标

证明玩家客户端可以从开局运行到至少一种胜利或失败，并且主要产品支柱全部可体验。

### 内容边界

- 1 个复合主场景；
- 2 个主要政体和 2-3 个次要政体；
- 约 12 个政治中心和 40-60 个治理地区；
- 约 50-80 名人物；
- 一个完整农业周期和干湿季转换；
- 自动会战，不包含手动节点会战。

### 退出 Gate

- 玩家客户端从新游戏到一种胜利或失败完整可玩；
- AI 可以在同一客户端规则下完成整局；
- 主要系统没有 fixture-only 或 no-op UI；
- 至少 5 次人工通关/失败记录，包含操作问题和系统问题；
- 存档、载入、重启、旧版本拒绝或迁移语义明确；
- 性能、内存、错误恢复和可访问性达到 Alpha 基线；
- 无已接受 P0/P1。

## 11. R6 - Content Alpha

### 目标

在产品闭环稳定后扩展历史启发内容，并验证内容生产成本、文化风险和重复可玩性。

### 交付

- 2-3 个目标场景；
- 人物、政体、事件、术语和地图来源登记；
- HISTORICAL、INFERRED、COMPOSITE、FICTIONAL 标签完整；
- 场景目标、AI 目标和事件后果通过统一内容 schema；
- 反滚雪球、继承冲击、臣属重谈和战后恢复调平；
- 本地化术语表和翻译 QA；
- 音频、美术使用资产槽位接入，不改变权威模拟。

### 退出 Gate

- 内容可以由管线构建、验证、回滚和差异审查；
- 重大文化、宗教、族群和暴力表达经过对应审查；
- 多场景长跑没有系统性崩溃或单一必胜策略；
- 内容生产没有绕过来源和审查流程；
- 产品所有者确认内容规模足以进入 Beta 准备。

## 12. R7 - Beta Content Lock

### 目标

冻结主要系统、交互、内容范围和平台目标，只保留缺陷修复和明确调平。

### 必须满足

- R1-R6 全部 `PASS`；
- Chrome、Edge、Firefox 和 Windows 矩阵完成；
- 1280x720 至 4K、100%-200% DPI 关键布局完成；
- 主要流程通过键盘和基础辅助技术检查；
- 内容、历史、文化、语言和本地化审查完成；
- 人工完整通关和新玩家首次体验完成；
- 存档迁移策略冻结；
- 无 P0/P1，P2 有明确所有者和处理决定；
- 产品所有者执行 Human Gate。

### Gate 结果

只允许：

```text
HUMAN_PASS
REQUEST_CHANGES
DEFER
```

核心流程缺失、真实运行时未闭环或地图方向未接受时，不得使用 `PASS_WITH_LIMITS` 绕过。

## 13. R8 - Release Candidate / 1.0

### 目标

证明候选构建可以真实发布和恢复，而不只是本地 smoke package。

### 交付与 Gate

- 可追溯 Web 和 Windows 构建；
- 安装、更新、卸载、备份和回滚；
- 内容 manifest、存档版本、构建版本和 release hash 对齐；
- CSP、Electron IPC、依赖、许可证、SBOM 和供应链审计；
- 崩溃诊断和隐私边界；
- 发布版多日长跑和人工通关；
- 商店资料、文化鸣谢、已知问题和支持文档；
- 正式发布渠道、商业模式、产品名和商标 Human Gate；
- 无 P0/P1，产品所有者最终 `HUMAN_PASS`。

## 15. Gate 证据规则

每个 Gate 必须同时包含：

1. **自动证据**：类型检查、单元、属性、集成、E2E、确定性、存档和性能；
2. **运行时证据**：证明生产入口实际执行目标路径；
3. **人工证据**：由未实现该功能的人完成指定流程；
4. **负向证据**：非法、重复、过期、资源不足、损坏和恢复场景；
5. **范围证据**：没有通过 fixture、debug 开关、测试 CSS 或旁路 API 伪造结果；
6. **审查证据**：领域 reviewer 和独立 QA 检查实际 diff 与运行结果。

以下证据不能单独支持产品 Gate：

- DOM 元素存在；
- `data-*` 属性符合预期；
- fixture 文本显示成功；
- Runner 在无头模式完成；
- Storybook story 可点击；
- 构建命令退出码为 0；
- agent 自己评价体验可用。

## 16. Gate 失败与限制规则

- P0/P1 必须阻止 Gate；
- 主要玩家流程缺失必须阻止 Gate；
- `PASS_WITH_LIMITS` 只允许不影响阶段定义的 P2/P3、外部工具限制或明确非目标；
- 限制必须有所有者、关闭条件和最迟处理阶段；
- 同一核心限制不得连续跨越两个产品 Gate；
- 产品所有者明确拒绝的方向必须进入 `REQUEST_CHANGES`，不能改名后再次提交；
- Gate 失败不回滚已验证底层资产，但必须回到最近满足定义的产品阶段。

## 18. 明确停止事项

在对应 Gate 前不得继续：

- R1 前：新的玩家功能、内容填充、美术生产、内容锁；
- R2 前：大规模 UI 改版和复杂工作台扩张；
- R3 前：正式地图内容、历史边界和生产地形资产；
- R5 前：大规模人物和事件填充；
- R7 前：正式内容锁；
- R8 前：商店提交、正式签名发布和商业承诺。

手动节点会战继续保持 `DEFER_MANUAL_NODE_BATTLE`，不阻塞 R0-R8。服务器、多人、遥测、任意代码 Mod、Rust/WASM、复杂海战和 3D 战斗继续不在 1.0 自动范围内。

## 19. Roadmap 成功判据

本 Roadmap 成功不是因为关闭了更多任务，而是因为依次证明：

1. 玩家命令真实改变权威世界；
2. 玩家知道现在发生了什么以及为什么；
3. 地图可以支持观察、计划和行动；
4. 治理与战争形成同一个反馈循环；
5. 玩家客户端可以从开局运行到终局；
6. 内容、文化、平台和发布质量在系统稳定后得到验证。

在这些证明完成前，项目可以拥有成熟的工程资产，但不得把自身描述为已经完成的 Beta 产品。
