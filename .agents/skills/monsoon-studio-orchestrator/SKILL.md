# Monsoon Studio Orchestrator

Treat the repository as the source of truth. This Skill turns a request into a controlled task graph and drives it through specialized Codex threads.

## Required reading

Before acting, read:

1. `AGENTS.md`
2. `PROJECT_STATUS.md`
3. `docs/11-roadmap.md`
4. `docs/14-multi-agent-operating-model.md`
5. the task-relevant system/technical documents
6. `references/message-protocol.md` and `references/routing-workflow.md` in this Skill

Run `git status`. Never overwrite uncommitted human work.

## Determine the mode

Classify the request as one of:

- `START_PROJECT`: establish or resume milestone/task graph.
- `EXECUTE_TASK`: implement a specific READY task.
- `REVIEW_TASK`: independently verify a REVIEW task.
- `RESEARCH`: produce evidence/claim records, not implementation.
- `BALANCE`: run controlled simulations and report distributions.
- `RELEASE`: run release gates and package artifacts.
- `AUTONOMOUS_GOAL_MODE`: continue dependency-ready tasks under `AUTONOMOUS-GOAL-MODE-001` gates after explicit product-owner approval.

If no task ID exists, the root/lead thread creates one. Child agents do not invent broader tasks.

## Core orchestration loop

1. **Load state**
   - Inspect `project/tasks/active`, `project/tasks/thread-registry.json`, recent `project/messages/outbox`, git branch/worktrees, and milestone exit criteria.
   - Initialize directories with `node .agents/skills/monsoon-studio-orchestrator/scripts/taskctl.mjs init` when absent.
   - On a fresh repository, seed the supplied task graph exactly once with `node .agents/skills/monsoon-studio-orchestrator/scripts/taskctl.mjs seed-initial`; the command is idempotent and skips existing task IDs.

2. **Create or validate task**
   - Ensure one primary writer, independent reviewer, allowed/forbidden paths, dependencies, acceptance criteria, required tests, risk level, and `route_to`.
   - Validate with `taskctl.mjs validate <TASK_ID>`.
   - R3/R4 or frozen-decision changes require ADR/research gate before implementation.

3. **Plan thread topology**
   - Keep `max_threads <= 6`, `max_depth = 1`, and reserve one review slot.
   - Parallelize only independent paths. Never spawn two writers for the same package/file area.
   - Select custom agents from `.codex/agents`; use the current model-routing matrix in `docs/26-model-routing-amendment.md`.
   - Use GPT-5.5 xhigh for lead, systems, gameplay, QA, and security judgment; GPT-5.5 high for complex implementation; GPT-5.4-mini for support analysis; GPT-5.3-Codex-Spark only for bounded mechanical execution.

4. **Spawn and register**
   - Explicitly call native `spawn_agent` for each needed role. Record returned thread IDs with:
     `taskctl.mjs set-thread <TASK_ID> <ROLE> <THREAD_ID>`.
   - Give each child: task ID, required docs, allowed/forbidden paths, inputs/commit, acceptance criteria, tests, risk, and required `ROUTE_TO`.
   - Do not paste the entire repository context; pass exact paths and a focused summary.

5. **Wait and collect**
   - Use `wait_agent` for expected completions. A child must return a structured handoff and write a JSON copy under `project/messages/outbox`.
   - A timeout/failure is `PARTIAL` or `BLOCKED`, never implicit success.

6. **Validate handoff**
   - Run:
     `node .agents/skills/monsoon-studio-orchestrator/scripts/validate-handoff.mjs <path>`.
   - Check stated files/commit/test commands against the workspace. Do not trust a bare “tests passed”.

7. **Route automatically**
   - Read `route_to` and `requested_action` from the validated handoff.
   - If that role has an active thread ID in the registry and the thread is still valid for the current model-routing state, call native `send_input` with the compact routing message generated by:
     `node .agents/skills/monsoon-studio-orchestrator/scripts/render-route-message.mjs <handoff-path>`.
   - If no active target thread exists, `spawn_agent` with the named custom role, register it, then `send_input` or include the route message in its initial prompt.
   - Normal routes:
     - designer → architect/test/engineer
     - writer → assigned reviewer, normally qa_reviewer or systems_architect
     - UI/Electron writer → test_engineer and/or security_reviewer before QA acceptance
     - historical scout → historical_researcher
     - balance analyst → gameplay_designer
     - reviewer REQUEST_CHANGES → original writer
     - reviewer ACCEPT → lead_orchestrator

8. **Review gate**
   - Only reviewer output beginning `ACCEPT` may advance task to `ACCEPTED`.
   - Spark results cannot be final acceptance. Route them to a GPT-5.5 high/xhigh reviewer.
   - `REQUEST_CHANGES` returns to the original writer through `send_input`; do not spawn a competing fixer unless the original thread is unavailable and the lead explicitly transfers ownership.
   - `BLOCK` pauses the task and routes to the named decision owner.

9. **Integrate and close**
   - Lead verifies combined gates, updates task/ADR/status, and records commit/integration result.
   - Use `close_agent` after accepted work or durable blocked handoff; do not keep stale threads open.
   - Archive the task only after integration. Thread IDs are session aids, not durable project state; a new root session may spawn a fresh role using persisted handoffs.

10. **Continue the graph**
   - Select the next dependency-satisfied READY task only within the current milestone.
   - Route its kickoff to the owner role. Never silently advance beyond a milestone exit gate.

## Autonomous Goal Mode

When `project/goal-mode-state.json` exists with `enabled: true`, load `docs/27-autonomous-goal-mode.md` before continuing the graph. The mode is a controlled delivery overlay; it does not change frozen decisions, the model matrix, parent-owned routing, reviewer separation, or task scope rules.

Autonomous continuation may proceed only when:

- `human_gate.required` is false;
- the next task is dependency-satisfied and inside the current milestone;
- WIP limits from `docs/14-multi-agent-operating-model.md` and `docs/27-autonomous-goal-mode.md` are available;
- the previous writer/reviewer handoffs have been validated;
- required tests and diff-scope gates are recorded;
- no stop condition from this Skill or `docs/27-autonomous-goal-mode.md` is active.

After each writer handoff, review result, or integration checkpoint, update `project/goal-mode-state.json` with the current task, active threads, ready tasks, unresolved risks, Human Gate state, and timestamp. If a restart occurs, recover from the newest validated handoff and current git state; do not continue from memory or stale thread IDs.

## Spark task policy

Spark is a fast executor, not an architect, designer, security judge, or final reviewer.

Spark may perform:

- explicit package scaffolding;
- mechanical config changes;
- boilerplate against approved interfaces;
- listed unit tests, fixtures, DTOs, validators, adapters;
- lint/format/typecheck fixes;
- small React/CSS changes;
- simple pnpm, CI, build, and check scripts.

Spark must not decide architecture, cross-package public boundaries, domain rules, save compatibility, deterministic strategy, security outcomes, unapproved production dependencies, large refactors, or final acceptance.

Every Spark task prompt must contain:

```text
TASK
CONTEXT
ALLOWED_PATHS
FORBIDDEN_PATHS
ACCEPTANCE_CRITERIA
REQUIRED_TESTS
STOP_CONDITIONS
REQUIRED_HANDOFF
ROUTE_TO
```

Spark stop conditions:

- public API change required;
- allowed paths are insufficient;
- multiple reasonable interpretations exist;
- design and implementation conflict;
- production dependency is needed;
- tests expose a cross-module defect;
- required tests cannot run;
- security, determinism, or data-loss risk appears.

On a stop condition, Spark emits `PARTIAL` or `BLOCKED` handoff and stops.

## Thread migration

- Threads with no useful handoff may be closed and recreated under the current model matrix.
- Active writers finish their current minimal atomic change and hand off; later stages use the new matrix.
- Tasks already in `REVIEW` use the new reviewer model and do not restart the writer.
- `ACCEPTED` tasks stay accepted; M0 gate QA may recheck them, but only real defects create follow-up tasks.
- Do not claim the current lead thread has switched model or effort unless the environment actually exposes that switch.

## Mandatory child handoff

Every child must produce fields defined in `references/message-protocol.md` and `project/schemas/handoff.schema.json`:

```text
task_id, message_id, from_role, status, summary, artifacts,
branch_or_worktree, commit, tests_run, decisions, risks, blockers,
route_to, requested_action, created_at
```

Use `assets/templates/handoff.json` as the starting shape.

## Stop conditions

Stop and route rather than guess when:

- a frozen decision would change;
- architecture/schema/save/RNG/time order is ambiguous;
- historical/cultural evidence is inadequate;
- a security or license risk appears;
- allowed paths are insufficient;
- another writer has uncommitted conflicting work;
- required tests cannot run.

## Final response to the human

Report only evidence-backed state:

- tasks accepted/partial/blocked;
- branches/commits/artifacts;
- exact tests run;
- decisions/ADRs;
- unresolved risks;
- next READY task and owner.

Never claim asynchronous future work. Complete the current orchestration turn with available tools and persist every unfinished state.
